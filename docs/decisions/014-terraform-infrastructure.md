# ADR 014: Terraform infrastructure as code

**Date:** 2026-02-27
**Status:** Accepted

## Context

The travel planner runs on Vercel with a Neon PostgreSQL database. Both were configured manually
via their respective dashboards. This creates a reproducibility problem — there is no
version-controlled record of what infrastructure exists, and recreating it after an incident
requires tribal knowledge.

A secondary concern is environment isolation. Vercel creates a preview deployment for every pull
request. Previously there was no mechanism to give those deployments their own database. A PR
containing a broken migration script would run `pnpm db:migrate` at Vercel build time against the
only Neon project that existed — the production one.

Atlantis was considered as the plan/apply automation tool but requires a persistent server
(~$10-15/month of EC2 or equivalent), which conflicts with the goal of minimising infrastructure
costs.

Terraform Workspaces were also considered and rejected. HashiCorp's own documentation notes that
workspaces are not designed for environment isolation — they share backend configuration and are
better suited to testing the same configuration with minor variable overrides. Separate root
modules with separate state give genuine isolation.

## Decision

Manage all infrastructure as Terraform code under `terraform/` using:

- **`kislydm/neon`** provider to manage the Neon Postgres project, database, and role.
- **`vercel/vercel`** provider to manage the Vercel project, GitHub integration, and all
  environment variables.
- **`hashicorp/random`** to generate `AUTH_SECRET` (eliminating manual `openssl rand` steps).
- **HCP Terraform free tier** as the remote state backend and VCS-driven CI/CD replacement for
  Atlantis. Connecting a GitHub repository to an HCP Terraform workspace provides speculative
  plans on PRs and apply-on-merge with zero server cost.

The codebase is organised into reusable modules and separate environment root modules:

```
terraform/
├── modules/
│   ├── neon-database/     # creates a Neon project, database, and owner role
│   └── vercel-project/    # creates a Vercel project with environment variables
└── environments/
    ├── production/        # HCP workspace: travel-planner-production
    └── preview/           # HCP workspace: travel-planner-preview
```

The `environments/preview` root module creates a dedicated Neon project for Vercel preview
deployments. The `environments/production` root module creates the production Neon project and
the Vercel project, configuring the `POSTGRES_URL` environment variable with different values per
Vercel target: `production` deployments use the production Neon connection string; `preview`
deployments use the preview Neon connection string.

Database migrations (`pnpm db:migrate`) run as part of the Vercel build command
(`pnpm db:migrate && pnpm build`). Drizzle's migrator is idempotent — it only applies unapplied
migrations and is safe to run on every deployment. If a migration script has a bug, the Vercel
preview build fails; production is not affected.

A lightweight `terraform-check` job is added to the existing GitHub Actions CI workflow. It runs
`terraform fmt -check` and `terraform validate -backend=false` on every pull request, requiring
no API credentials. Full speculative plans (which do require credentials) are handled by HCP
Terraform's own VCS integration.

## Consequences

**Positive:**
- All infrastructure changes are code-reviewed via pull requests and subject to speculative plans
  before being applied.
- The entire stack (Neon database, Vercel project, environment variables) can be reproduced from
  code after a catastrophic failure.
- `AUTH_SECRET` is generated by Terraform on first apply and stored only in HCP Terraform state
  and Vercel — it is never handled manually.
- PR preview deployments use an isolated database, so migration bugs fail the preview build
  without any risk to production data.
- Adding a new environment (e.g. staging) requires creating one new directory under
  `environments/` and one new HCP Terraform workspace — no changes to the shared modules.

**Negative / trade-offs:**
- The `branch.database_name` and `branch.role_name` arguments inside the `neon_project` resource
  cannot be changed after creation without recreating the project (destroying all data). These are
  set once at initial `terraform apply`.
- Applying preview before production is a manual bootstrap step. The preview `postgres_url` output
  must be copied into the production workspace as a variable. This is a one-time operation.
- A single shared preview Neon project is used for all PR preview deployments. If two PRs open
  simultaneously with conflicting migrations, they will conflict on the preview database. For a
  personal project with one active contributor this is not a practical concern.
