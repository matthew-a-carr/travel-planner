# Tech Debt Register

> This file captures known technical debt. Items are added during
> feature implementation when deviations from a spec cannot be resolved
> immediately. Items are removed (moved to Resolved) when the debt is
> paid down.
>
> **Review cadence:** Before planning any new spec, review this register.
> If any outstanding items are relevant to the new feature or can be
> addressed alongside it, include them in the spec's implementation order.
> Additionally, this register should be reviewed periodically (weekly or
> on-demand) to identify items that warrant their own spec.

| TD-009 | 2026-05-31 | mobile-e2e | **`mobile-e2e` failed on every dependency PR (#121/#129/#131) — ROOT-CAUSED: stale native-build cache, not flakiness.** Initially read as cold-sim "flakiness" (timeouts + an "App crashed on launch" mode) and mitigated with the `extendedWaitUntil(60s)` de-flake (#127) + a 3× retry loop (#130). Those were band-aids. **Real cause:** the job caches `apps/mobile/ios` (Pods + DerivedData) keyed on `hashFiles(pnpm-lock, package.json, app.json)` with `restore-keys: mobile-e2e-macOS-`, and prebuild had dropped `--clean`. Any dependency PR changes the lockfile → exact key MISS → `restore-keys` restores a stale `ios/` from a *different* dependency graph → the non-`--clean` incremental prebuild builds on that stale state → broken app that crashes/hangs on launch. `main` passes (exact key hit). Confirmed in #129's log: `Cache hit for restore-key: …` (prefix fallback, not exact). The retries couldn't help — the bad `.app` is built once. **Fix (this entry's PR):** prebuild now runs `--clean` whenever `cache-hit != 'true'` (key miss / restore-key fallback), so dependency PRs get a fresh build matching their deps; exact hits stay incremental/fast. Retries + `ios-crash-diagnostics` upload retained as insurance for genuine transients. **Pending:** verify the dep PRs (#129/#131) go green once rebased on the fix, then resolve. | Medium | unowned |
